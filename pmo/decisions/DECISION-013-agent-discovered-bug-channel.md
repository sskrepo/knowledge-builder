# DECISION-013: Agent/Architect-Discovered Defect Channel in ADB

**Status**: DECIDED
**Date**: 2026-05-16
**Decided by**: User
**Informed by**: 2026-05-16 hardening session — ~11 proactive defects found+fixed by agents/architect with no ADB record; DECISION-008 gap analysis
**Amends**: DECISION-008 (adds a third discovery channel; does not change the single-source-of-truth decision)

---

## Context

DECISION-008 established ADB (`KBF_BUG_REPORTS` + `KBF_AUDIT_RUNS`) as the single source of truth for all bug records. It defined two write paths:

1. **User-reported** — filed via `reportBug` MCP tool → `AdbErrorStore.record_user_bug()` → `KBF_BUG_REPORTS`
2. **Critic-found** — filed by `reviewSkillSession` → `KBF_AUDIT_RUNS`

During the 2026-05-16 hardening session, approximately 11 defects were discovered proactively by agents and the architect — found during investigation, root-cause analysis, or e2e verification — and fixed in the same commit. Because DECISION-008 defined no channel for these discoveries, they were committed to git with detailed RCA in the commit message but were **never filed into ADB**. This creates a gap: ADB is not a complete record of known defects; downstream bug bash (`kbf_ops.bugBash`) and `kb-cli export-bugs` miss these fixes entirely.

---

## Options Considered

### Option A — File into `KBF_BUG_REPORTS` with a discovery discriminator (CHOSEN)

- Call `AdbErrorStore.record_user_bug(entry)` for agent/architect discoveries
- Add `discovered_by` ∈ `{agent, architect, user}` as an `extra_json` key (no schema migration required — `record_user_bug` already JSON-serializes all non-canonical keys into the `extra_json` CLOB)
- Also carry `status`, `fix_commit`, `severity`, `session`, `root_cause` in `extra_json`
- `queue_id` keeps the `BUG-queue-<uuid4 hex[:5]>` convention — single namespace, queryable alongside user reports, distinguishable by `discovered_by`
- `pmo/bugs/*.md` stays generated (`kb-cli export-bugs`); no hand-written files

**Pros**: ADB stays the single and complete source of truth. No schema change. Discoverable by bug bash. `discovered_by` field cleanly distinguishes user / agent / architect finds. Consistent with DECISION-008.

**Cons**: `KBF_BUG_REPORTS` was conceptually "user reports" — now slightly broader. Mitigated by `discovered_by` discriminator and this DECISION recording the extension.

### Option B — New table `KBF_PROACTIVE_FIXES`

- Separate table for agent/architect-discovered defects
- Bug bash would need to query two tables
- Schema migration required

**Cons**: Splits the bug record. `bugBash` tool must query two tables. No benefit over Option A's discriminator pattern.

### Option C — Git commits are sufficient record

- No ADB write; commit message + RCA in git is the authoritative record
- Incompatible with DECISION-008 (ADB-only query surface); bug bash misses these bugs

**Cons**: Directly violates DECISION-008 and the "ADB is single source of truth" principle.

---

## Decision

**Option A: File agent/architect-discovered defects into `KBF_BUG_REPORTS` via `record_user_bug` with a `discovered_by` discriminator.**

### Entry shape for agent/architect discoveries

```python
entry = {
    # Canonical KBF_BUG_REPORTS columns:
    "request_id":  f"arch-rca-{sha8}",          # or agent-rca-{sha8}
    "queue_id":    f"BUG-queue-{uuid4().hex[:5]}",
    "timestamp":   "<commit author date ISO>",    # not filing date
    "tool":        "<most relevant MCP tool>",
    "description": "<symptom + reference to related user report if any>",

    # extra_json keys (no schema change needed):
    "discovered_by": "architect" | "agent",       # never "user"
    "status":        "fixed",
    "fix_commit":    "<SHA>",                      # or list of SHAs for clusters
    "severity":      "HIGH" | "Medium" | "Low",
    "session":       "<session-id or date label>",
    "root_cause":    "<concise RCA from commit diff/message>",
}
```

**Severity rule** (project standard): any silent wrong-output or silent degradation = HIGH regardless of user visibility. Medium = functional defect, not silent. Low = test/tooling only.

---

## Consequences

- `KBF_BUG_REPORTS` is now a complete defect record covering user-reported, critic-found (via `KBF_AUDIT_RUNS`), and agent/architect-proactively-discovered bugs. The `discovered_by` field distinguishes channels.
- `pmo/bugs/*.md` snapshots generated by `kb-cli export-bugs` will include agent/architect entries. They are still read-only exports — not the source of truth.
- `kbf_ops.bugBash` reads from ADB only and will now see the complete picture.
- The JSONL dual-write (`~/.kbf/store/user_bugs.jsonl`) also receives these entries — `kb-cli watch-bugs` dedup remains valid.

---

## Standing Practice (going forward)

Every fix to a framework defect **must** include an ADB bug record filed (via `record_user_bug` or the backfill pattern) in the same change as the code. The record must carry:
- `discovered_by` — who found it
- `fix_commit` — the fixing SHA
- `root_cause` — concise RCA (one–three sentences)
- `status: "fixed"`

This is non-negotiable. A fixed defect with no ADB record is incomplete work.
