Great question — it's neither, technically. Let me draw the distinction clearly because I've been using overlapping vocabulary.

Three different things — don't confuse them

│  PERSONA KNOWLEDGE BUILDER  (write-side, ingestion)                         │
│  - A YAML config + the framework's pipeline that processes it               │
│  - Runs on schedule (cron) + on webhook events                              │

Why a Knowledge Builder is NOT an agent
Even though it uses LLM calls (in the parser), it doesn't make autonomous decisions:

It doesn't decide which sources to ingest — the YAML config tells it
It doesn't decide what fields to extract — the JSON Schema tells the LLM
It doesn't loop or retry creatively — failure → log + retry on schedule
It doesn't call other agents
It's just: "for each new item from these sources, parse it with this schema, write it to this store." That's a pipeline.

Why it's NOT a skill either
A skill is called synchronously by an orchestrator on demand. The Knowledge Builder isn't called by anything in the read path — it runs on its own clock, populating the stores so the read path has data to retrieve.

What about the persona-knowledge-builder name?
Yes — the name is slightly overloaded. Looking at our docs (docs/wiki/persona-knowledge-builder.md), what we call a "Knowledge Builder" is really:

The YAML config (declarative spec of what to ingest)
The JSON Schema (declarative spec of what to extract)
The gold set (declarative spec of how to evaluate)
The framework's runtime then executes that spec via the ingestion pipeline. So "Knowledge Builder" is closer to "build spec" or "ingestion blueprint" than to a runtime entity.

If we wanted to rename for clarity:

Today: "PM Knowledge Builder" → ambiguous (config? agent? skill?)
Clearer: "PM Ingestion Spec" + the framework runs it as a job
I'd keep the current name though — the user community is already used to "Knowledge Builder" and renaming now would cost more than it gains. Just remember it's a specification, not a running agent.
How this maps to your TPM example
When you said "can I take existing code, edit a little of TPM knowledge base 'what to extract' part, and start creating knowledge base for TPM?" — the literal mechanics:

You edit:
  framework/persona_builders/tpm.yaml             ← what sources, what KBs
  framework/parsers/schemas/tpm/*/v1.json         ← what to extract
Then run (once Phase 1 lands):
  $ kb-cli validate framework/persona_builders/tpm.yaml      ← lint
  $ kb-cli ingest --dry-run --sample 5 ...                   ← preview parse on 5 items
  $ kb-cli eval framework/persona_builders/tpm.yaml          ← run TPM gold set
  $ kb-cli promote framework/persona_builders/tpm.yaml       ← flip to production
That's it. The framework's ingestion pipeline reads tpm.yaml, fan-outs to
configured sources, parses with the schema, writes to the configured stores.
No new code. No new agent. No skill change. Just the existing pipeline
applied to new config.
The TPM persona context skill (framework/persona_skills/tpm.py) is a separate thing — a 50-line file that's already stubbed; it gets activated automatically once tpm.yaml flips to status: production and its KBs have data.

Short answer
The Persona Knowledge Builder for TPM = a YAML config + JSON schemas + gold set. The framework's ingestion pipeline executes it as a scheduled job. It's not a skill, not an agent — it's an ingestion specification.
