AIRA Vector Search

Detailed Explanation of How AIRA Stores and Retrieves KB Data

A DB-verified, plain-English walkthrough of AIRA's current knowledge-base storage, vector generation, retrieval flow, schema, and limitations.

Storage

Readable KB records live in service-specific dataset tables.

Embedding

The DB converts chunk text into CHUNK_EMBEDDING.

Search

KB_VECTOR_SEARCH compares the query vector with stored vectors.

Answer

GenAI reads retrieved chunks and writes RCA/resolution.

Description

AIRA keeps a knowledge base of past resolved Jira tickets. Each entry says what error happened, what component was involved, what the root cause was, how it was resolved, and which ticket it came from.

When a new issue arrives, AIRA searches these historical entries, retrieves the most relevant chunks, and gives those chunks to GenAI so it can produce a likely root cause, resolution, and similar ticket reference.

End-to-End Flow

1. A past Jira ticket is resolved.
2. DataPipelineJob reads that ticket.
3. GenAI extracts a clean KB record.
4. A validation agent checks duplicate/validity.
5. AIRA inserts the record into <SERVICE>_DATASETS.
6. AIRA calls BATCH_INSERT_DATASETS_VECTORS.
7. The DB creates an embedding in <SERVICE>_DATASETS_VECTOR.
8. A user asks AIRA about a new error.
9. AIRA calls KB_VECTOR_SEARCH.
10. The DB returns similar historical chunks.
11. Java filters and dedupes the chunks.
12. GenAI reads the chunks and generates RCA/resolution.

What AIRA Stores

The main KB payload is stored in CHUNK_TEXT as JSON.

{
  "TicketId": "ADP-246527",
  "Component": "Source - adp-gi-domu-chain-patching",
  "Error Message": "NullPointerException: Pod DB state was null during resource validation",
  "Root Cause": "Pod db status was NULL; rotate pod db workflow set the pod db to NULL state.",
  "Resolution": "Updated pod db state to ACTIVE/None."
}

AIRA also stores row metadata such as DATASET_ID, CHUNK_INDEX, COMPONENT, JIRA_CREATION_DATE, STACK, ASSIGNEE, and agent validation fields.

Chunking Logic

Before AIRA inserts a KB record into the dataset table, Java converts the record to JSON and splits it into chunks.

String jsonText = JsonUtil.extractJson(recordDetails.dataPipelineRecord().toString());
List<String> chunks = chunkContent(jsonText);

The chunking helper is TextProcessingUtil.chunkContent(...). Its current maximum chunk size is:

MAX_CHUNK_SIZE = 2000 characters

The helper tries to split on sentence boundaries first. It uses this pattern:

(?<=\.)\s+

That means it prefers to split after a period followed by spaces, so it avoids cutting normal sentences in the middle when possible.

Example: Small KB Record

If the full JSON is less than 2000 characters, AIRA stores one row:

DATASET_ID   CHUNK_INDEX   CHUNK_TEXT
ADP-100      0             { whole KB JSON }

Example: Large KB Record

If the JSON is larger than 2000 characters, AIRA creates multiple rows for the same ticket:

DATASET_ID   CHUNK_INDEX   CHUNK_TEXT
ADP-100      0             first part of KB JSON
ADP-100      1             second part of KB JSON
ADP-100      2             third part of KB JSON

If One Sentence Is Too Long

If one sentence itself is longer than 2000 characters, AIRA splits that long sentence into 2000-character pieces.

one 4500-character sentence
-> chunk 0: chars 0-1999
-> chunk 1: chars 2000-3999
-> chunk 2: chars 4000-4499
Simple way to remember it: one ticket can become many chunk rows. DATASET_ID is the ticket id, and CHUNK_INDEX is the chunk number for that ticket.

Insert Flow

VectorDbHelper.addDataset(...) inserts KB records with a MERGE so the same ticket/chunk is not inserted twice.

MERGE INTO <dataset_table> dataset
USING (SELECT ? AS dataSet_id, ? AS chunk_index FROM dual) src
ON (dataset.dataSet_id = src.dataSet_id AND dataset.chunk_index = src.chunk_index)
WHEN NOT MATCHED THEN
INSERT (
  dataSet_id, chunk_index, chunk_text, component,
  is_agent_created, is_human_validated, agent_comment,
  jira_creation_date, stack, assignee
)
VALUES (...)

After the insert, Java calls:

CALL BATCH_INSERT_DATASETS_VECTORS(<dataset_table>, <dataset_table>_VECTOR)

Vector Generation

BATCH_INSERT_DATASETS_VECTORS copies missing dataset rows into the vector table and generates CHUNK_EMBEDDING from CHUNK_TEXT.

DBMS_VECTOR.UTL_TO_EMBEDDING(
  chunk_text,
  JSON('{
    "provider": "OCIGenAI",
    "credential_name": "OCI_VECTOR_CRED",
    "url": "https://inference.generativeai.eu-frankfurt-1.oci.oraclecloud.com/20231130/actions/embedText",
    "model": "openai.text-embedding-3-large"
  }')
)

The procedure skips vector rows that already exist for the same DATASET_ID and CHUNK_INDEX.

Retrieval

Java calls the DB function with a raw query shaped like:

<projectKey>:- <stack>:- <user query>

Example:

FAAASP2T:- prod:- NullPointerException: Pod DB state was null during resource validation

The function returns:

DOCID
BODY
SCORE

Query Vector Details

When AIRA receives a new query, KB_VECTOR_SEARCH creates a vector for that query inside the database.

query_vec := DBMS_VECTOR.UTL_TO_EMBEDDING(p_query, JSON(...));

Today the function embeds the full raw query string, including project and stack:

FAAASP2T:- prod:- NullPointerException: Pod DB state was null during resource validation

That query vector is then compared against every CHUNK_EMBEDDING row in the selected vector table.

VECTOR_DISTANCE(CHUNK_EMBEDDING, query_vec)

The DB turns distance into a score:

1 / (1 + VECTOR_DISTANCE(CHUNK_EMBEDDING, query_vec))
DistanceMeaningScore effect
SmallQuery and chunk are similarHigher score
LargeQuery and chunk are less similarLower score
Simple way to remember it: the query gets one vector, every stored chunk already has a vector, and the DB compares the query vector with chunk vectors to rank the best matches.

KB_VECTOR_SEARCH

The function parses project and stack from the raw query, chooses a vector table, embeds the full raw query, and compares it with CHUNK_EMBEDDING.

1 / (1 + VECTOR_DISTANCE(CHUNK_EMBEDDING, query_vec))

For prod or demo, rows with STACK prod/demo keep full score; other stacks get a 0.90 multiplier. Stack is a preference, not a hard filter.

Actual DB Function

This is the live database function shape used today. Read it in four parts: parse the query, choose the vector table, create the query vector, then run vector-distance SQL.

What It Does In Simple Steps

  1. Reads p_query, expecting text like FAAASP2T:- prod:- error text.
  2. Extracts project_name and stack_name.
  3. Chooses one vector table, such as FAAASP2T_DATASETS_VECTOR.
  4. Creates query_vec using DBMS_VECTOR.UTL_TO_EMBEDDING.
  5. Compares query_vec against every row's CHUNK_EMBEDDING.
  6. Returns the best matching rows as DOCID, BODY, and SCORE.
Current behavior to note: top_k is passed into the function but the SQL currently uses hard-coded 15. Also, DOCID is returned as only DATASET_ID, so Java deduplication can collapse multiple chunks from the same ticket. 
CREATE OR REPLACE FUNCTION kb_vector_search (
    p_query IN VARCHAR2,
    top_k   IN NUMBER
) RETURN SYS_REFCURSOR IS
    v_results     SYS_REFCURSOR;
    query_vec     VECTOR;
    project_name  VARCHAR2(128);
    table_name    VARCHAR2(200);
    v_sql_stmt    VARCHAR2(4000);
    sep_pos       NUMBER;
    first_sep     NUMBER;
    second_sep    NUMBER;
    stack_name    VARCHAR2(50);
    v_age_expr    VARCHAR2(500);
BEGIN
    first_sep  := INSTR(p_query, ':-');
    second_sep := INSTR(p_query, ':-', first_sep + 1);

    IF first_sep = 0 THEN
        project_name := 'ADP';
        stack_name   := 'preprod';
    ELSE
        project_name := UPPER(TRIM(SUBSTR(p_query, 1, first_sep - 1)));

        IF second_sep = 0 THEN
            stack_name := 'preprod';
        ELSE
            stack_name := LOWER(TRIM(SUBSTR(p_query, first_sep + 2, second_sep - first_sep - 2)));
            IF stack_name IS NULL OR stack_name = '' THEN
                stack_name := 'preprod';
            END IF;
        END IF;
    END IF;

    IF project_name IN ('FAAASPATCH', 'FAAASTOOLS', 'FAAASUPG') THEN
        table_name := 'HCP_DATASETS_VECTOR';
    ELSIF project_name IN ('FAAAS') THEN
        table_name := 'FAAAS_DATASETS_VECTOR';
    ELSIF project_name IN ('FAAASINT') THEN
        table_name := 'FAAASINT_DATASETS_VECTOR';
    ELSIF project_name IN ('FAAASP2T') THEN
        table_name := 'FAAASP2T_DATASETS_VECTOR';
    ELSIF project_name IN ('SFAAS') THEN
        table_name := 'SFAAS_DATASETS_VECTOR';
    ELSIF project_name IN ('FAAASPROV') THEN
        table_name := 'FAAASPROV_DATASETS_VECTOR';
    ELSIF project_name IN ('ADBMIGRATION') THEN
        table_name := 'ADBSMIG_DATASETS_VECTOR';
    ELSIF project_name IN ('FAAASHCP') THEN
        table_name := 'FAAASHCP_DATASETS_VECTOR';
    ELSIF project_name IN ('FAAASOPS') THEN
        table_name := 'FAAASOPS_DATASETS_VECTOR';
    ELSE
        table_name := 'ADP_DATASETS_VECTOR';
    END IF;

    query_vec := DBMS_VECTOR.UTL_TO_EMBEDDING(
        p_query,
        JSON('{
            "provider": "OCIGenAI",
            "credential_name": "OCI_VECTOR_CRED",
            "url": "https://inference.generativeai.eu-frankfurt-1.oci.oraclecloud.com/20231130/actions/embedText",
            "model": "openai.text-embedding-3-large"
        }')
    );

    v_age_expr := 'GREATEST(SYSDATE - COALESCE(CAST(JIRA_CREATION_DATE AS DATE), CAST(TIME_CREATED AS DATE)), 0)';

    IF stack_name IN ('prod', 'demo') THEN
        v_sql_stmt :=
        'SELECT DATASET_ID AS DOCID,
                BODY,
                TRUNC(1 / (1 + VECTOR_DISTANCE(CHUNK_EMBEDDING, :b1)), 3) AS RAW_SCORE,
                TRUNC(
                    (1 / (1 + VECTOR_DISTANCE(CHUNK_EMBEDDING, :b1))) *
                    CASE
                        WHEN STACK IN (''prod'', ''demo'') THEN 1.00
                        ELSE 0.90
                    END
                , 3) AS SCORE
         FROM ' || table_name || '
         ORDER BY SCORE DESC,
                  NVL(' || v_age_expr || ', 3650) ASC
         FETCH FIRST :b3 ROWS ONLY';

        OPEN v_results FOR v_sql_stmt
        USING query_vec, query_vec, 15;
    ELSE
        v_sql_stmt :=
        'SELECT DATASET_ID AS DOCID, BODY, ' ||
        'TRUNC(1 / (1 + VECTOR_DISTANCE(CHUNK_EMBEDDING, :b1)), 3) AS SCORE ' ||
        'FROM ' || table_name || ' ' ||
        'WHERE DATASET_ID IN ( ' ||
        ' SELECT DATASET_ID ' ||
        ' FROM ' || table_name || ' ' ||
        ' ORDER BY TRUNC(1 / (1 + VECTOR_DISTANCE(CHUNK_EMBEDDING, :b1)), 3) DESC, ' ||
        ' NVL(' || v_age_expr || ', 3650) ASC' ||
        ' FETCH FIRST :b2 ROWS ONLY ' ||
        ') ' ||
        'ORDER BY TRUNC(1 / (1 + VECTOR_DISTANCE(CHUNK_EMBEDDING, :b1)), 3) DESC, ' ||
        'NVL(' || v_age_expr || ', 3650) ASC, ' ||
        '1 / (1 + VECTOR_DISTANCE(CHUNK_EMBEDDING, :b1)) DESC';

        OPEN v_results FOR v_sql_stmt
        USING query_vec, query_vec, 15, query_vec, query_vec;
    END IF;

    RETURN v_results;
END;

Suggested Fixes Later

1. Use top_k instead of hard-coded 15.
2. Return DATASET_ID || ':' || CHUNK_INDEX AS DOCID if Java should keep multiple chunks per ticket.
3. Embed only the actual question text, not the full project/stack prefix.
4. Add component, failureContext, and tag matching after PROPERTIES is added.

Java Filtering

ContextBuilderNode applies these rules after DB retrieval:

minimum score = 0.50
sort = score descending
dedupe = docId
max context size = 50,000 characters

Each chunk is formatted and sent to GenAI as part of the knowledge-base context.

Final Answer

GenAI receives the retrieved KB chunks and the user query. It writes output using these sections:

Root_Cause:
Resolution:
Similar ticket for reference:

Java parses those into probableRootCause, probableResolution, and similarTicketReference.

What Is Missing Today

AIRA does not currently store or use these structured fields in DB vector retrieval:

failureContext
failureType
errorFamily
resourceType
operationArea
failureStage

Today the system mostly relies on vector similarity plus light stack preference.

Current Limitations

  • No PROPERTIES column exists today.
  • No structured failure context or tag matching exists in DB retrieval.
  • No component filter exists in KB_VECTOR_SEARCH.
  • Stack is a boost/penalty, not a hard filter.
  • top_k is accepted by the function, but live SQL currently uses hard-coded 15.
  • The query embedding includes the service/stack prefix along with the actual error text.

Recommended Future Improvement

Add a JSON property bag to each base dataset table:

ALTER TABLE <SERVICE>_DATASETS ADD (
  PROPERTIES CLOB CHECK (PROPERTIES IS JSON)
);

Recommended shape:

{
  "failureContext": "rotate pod db / resource validation",
  "tags": {
    "failureType": "null-state",
    "errorFamily": "null-pointer",
    "resourceType": "pod-db",
    "operationArea": "workflow",
    "failureStage": "resource-validation"
  }
}
In simple terms, PROPERTIES gives AIRA structured clues about where and why the failure happened.

Tag Matching

Tags should be secondary support, not the primary match. Hard gates should be same service/table, same or compatible component, same or compatible failure context, and similar error message.

TagSuggested weight
errorFamily0.30
failureType0.25
resourceType0.20
operationArea0.15
failureStage0.10

Tags help choose between close matches, but they should not make an unrelated ticket look relevant.

Simple Example

Incoming query:
FAAASP2T:- prod:- NullPointerException: Pod DB state was null during resource validation

Today:
1. DB chooses FAAASP2T_DATASETS_VECTOR.
2. DB embeds the raw query.
3. DB compares it with CHUNK_EMBEDDING values.
4. DB returns top similar BODY chunks.
5. Java keeps chunks with score >= 0.50.
6. GenAI reads those chunks and writes RCA/resolution.

Final Summary

Today AIRA's KB retrieval is service-specific table selection, vector similarity, light stack preference, Java score filtering, and GenAI answer generation.

It is not yet component-aware, failure-context-aware, tag-aware, or strictly stack-filtered. The next improvement should be adding PROPERTIES and using it in KB_VECTOR_SEARCH as compatibility filtering and scoring support.
