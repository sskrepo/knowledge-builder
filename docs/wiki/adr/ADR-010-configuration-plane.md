# ADR-010 ‚Äî Configuration plane

## Status
Accepted (2026-05-05).

## Context
The framework has many infra dependencies (Oracle 23ai ADB, OCI Vault/Object Storage/Streaming/Functions/Compute, OpenAI, Confluence, Jira, Git, UDAP). A user-facing configuration plane is needed so deployments are infra-as-config, secrets stay in Vault, and code never has to change to point at a different env or source.

## Decision

### Three-tier hierarchy
```
framework/config/
‚îú‚îÄ‚îÄ _schema.json                  # JSON-Schema validating every env file
‚îú‚îÄ‚îÄ dev.yaml                      # one per env
‚îú‚îÄ‚îÄ staging.yaml
‚îú‚îÄ‚îÄ prod.yaml
‚îú‚îÄ‚îÄ adapters/                     # adapter-type defaults, env-overridable
‚îÇ   ‚îú‚îÄ‚îÄ confluence.yaml
‚îÇ   ‚îú‚îÄ‚îÄ jira.yaml
‚îÇ   ‚îú‚îÄ‚îÄ git.yaml
‚îÇ   ‚îú‚îÄ‚îÄ udap.yaml
‚îÇ   ‚îî‚îÄ‚îÄ openai.yaml
‚îî‚îÄ‚îÄ shim_faaas.yaml               # FAaaS domain ontology, env-independent
```
Plus `framework/persona_builders/{persona}.yaml` (per ADR-004) that reference adapters and Vault paths.

### Tier 1 ‚Äî environment config
`config/{env}.yaml` holds connection details, OCIDs, region, ADB schema map, OpenAI provider settings, observability targets. **Sensitive values are `vault://` references; never plaintext.**

Selected via env var `KBF_ENV=dev|staging|prod` at process start.

### Tier 2 ‚Äî adapter config
`config/adapters/{kind}.yaml` holds source-type defaults: base URLs, default Vault auth-secret refs, pagination, rate limits, webhook receivers, mode switches (per ADR-011 for Confluence/Jira).

Adapter configs are **env-aware via overlay**: env config can override any adapter field with `adapters.{kind}.<key>: value`.

### Tier 3 ‚Äî domain ontology
`config/shim_faaas.yaml` is the FAaaS world model (per ADR-006). Same in every env; loaded into orchestrator's prompt at startup.

### Loading order at process start
1. Read `KBF_ENV` env var (default: `dev`).
2. Load `config/{env}.yaml`.
3. Load each `config/adapters/*.yaml`; apply env overlays.
4. Load `config/shim_faaas.yaml`.
5. Validate combined config against `_schema.json`. Fail fast on schema violations.
6. Resolve every `vault://` reference. Fail fast if any required secret is missing.
7. Capability-probe every adapter (e.g., MCP servers per ADR-011). Fail fast on missing capabilities.
8. Load every `persona_builders/*.yaml`; aggregate `kb_card`s into `shim_kb`.
9. Open ADB connection pool (one per schema in env config).
10. Service ready.

### Vault contract
- Every reference is `vault://kb/{slug}` (Vault path under the framework's prefix).
- `framework/vault_client.py` provides `resolve(ref) ‚Üí str` with a 60s in-memory cache and rotation on signal.
- Required-secrets manifest auto-generated by `framework/scripts/check-config.py`: emits the list of `vault://` strings every config (env + adapters + persona builders) references. Used by `bootstrap-vault.sh` for one-time setup.

### Configuration files vs. code rule
- **Users edit config**, never code.
- New OCID/URL/secret ‚Üí edit `config/{env}.yaml` or set new Vault entry.
- New persona/source ‚Üí edit `persona_builders/{persona}.yaml`.
- New adapter kind or new framework primitive ‚Üí code change (Architect PR + ADR if cross-cutting).

### Required-secrets manifest (auto-derived)
After loading configs, framework renders a manifest like:
```
vault://kb/adb-admin-dev
vault://kb/openai-api-key
vault://kb/confluence-readonly
vault://kb/confluence-mcp-token
vault://kb/jira-readonly
vault://kb/jira-mcp-token
vault://kb/git-readonly
vault://kb/udap-jdbc-url
vault://kb/udap-user
vault://kb/udap-password
vault://kb/confluence-webhook-secret
vault://kb/jira-webhook-secret
vault://kb/kb-incidents-rw-dev
vault://kb/kb-wiki-rw-dev
vault://kb/kb-code-rw-dev
vault://kb/kb-fa-rw-dev
vault://kb/kb-shim-rw-dev
```
`bootstrap-vault.sh` reads this manifest and walks the user through populating each entry interactively (prompts only for non-existent entries).

### Multi-env strategy
- Three env files: `dev`, `staging`, `prod`.
- `staging` mirrors `prod` topology with smaller resources for parity testing.
- Identical schema; differences are values only.
- v1 ships all three; deployment automation can flip between them via `KBF_ENV`.

### Observability touchpoints in config
- `observability.cost_telemetry_table` ‚Äî where cost rows land
- `observability.log_level`
- `eval.baseline_storage` ‚Äî OCI Object Storage path
- `eval.artifact_storage`
- `eval.ci_token_cap_per_pr_usd` ‚Äî hard cap on eval CI runs

### Validation tooling
- `framework/scripts/check-config.py`:
  - Validates env config against `_schema.json`
  - Checks every adapter config for known `mode` values
  - Verifies every `vault://` resolves (or lists missing)
  - Probes ADB connectivity with the configured wallet
  - Probes OpenAI with a 1-token request
  - Probes any MCP-mode adapters for required capabilities
  - Emits a green/red report

Run before every deploy. CI runs it on PRs that touch `config/` or `persona_builders/`.

## Considered alternatives
- **Single flat `config.yaml`** ‚Äî rejected; can't separate env, adapter, domain concerns.
- **Per-deploy env vars only (12-factor)** ‚Äî rejected; Vault references and structured ontology are awkward in env vars.
- **No domain ontology in config (just code)** ‚Äî rejected per ADR-006; needs to be diffable + PR-reviewable.
- **Keeping persona builders inside env config** ‚Äî rejected; persona configs are domain artifacts owned by persona teams.

## Consequences
- Bootstrap a new env = one YAML file + a Vault populate script.
- Persona team contributors don't touch infra; they edit `persona_builders/`.
- Architect contributors don't touch persona schemas; they edit `shim_faaas.yaml` and ADRs.
- `check-config.py` becomes a primary "is this thing healthy" tool ‚Äî runs before every deploy and in CI.
